# PRD — Incident Attribution & Evidence Management Suite

문서 버전: v1.0  
작성일: 2026-03-15  
문서 상태: Draft for Implementation

---

## 1. 제품 개요

본 제품은 자사 서비스/도메인에 대한 비정상 접근, 데이터 열람·반출 시도, 관리자 경로 접근, 인증 우회 정황 등을
다중 로그 소스에서 수집·정규화·상관분석하여 다음을 제공한다.

1. 사건 탐지 및 검색
2. 내부 귀속(`IP -> PC -> 계정/사용자`) 분석
3. 증거 고정 및 해시 관리
4. 사건 요약서/증거목록/고소장 초안 생성
5. 법무·보안·개인정보 대응 워크플로우

이 제품은 **공격 도구가 아니라 내부 조사·증거관리·문서화 도구**이다.

---

## 2. 문제 정의

현행 수기 대응 방식은 다음 문제를 가진다.

- 웹/WAF/AD/VPN/EDR/DHCP/NAT 로그가 분산되어 있어 조사 시간이 길다.
- IP 하나만으로는 실제 단말과 사용자를 특정하기 어렵다.
- 증거 수집 형식이 일관되지 않아 사건 제출 품질이 낮다.
- 원본 로그, 해시, 문서 산출물의 연결이 약해 증거능력 관리가 어렵다.
- 고소장/신고서 초안 작성에 사람 시간이 많이 든다.
- 외부 공인 IP 사건과 내부 자산 사건의 귀속 수준이 다름에도 같은 수준으로 취급되는 문제가 있다.

## 3. 목표

### 3.1 제품 목표
- 조사자가 IP/계정/단말/도메인 기준으로 사건을 즉시 검색할 수 있게 한다.
- 내부 사건은 `IP -> PC -> 주사용자/실제 로그인 사용자` 연결을 자동화한다.
- 외부 사건은 `성명불상 + 추가 조회 필요` 상태로 문서화한다.
- 사건별 증거 패키지와 해시 목록을 자동 생성한다.
- 고소장 초안 작성 시간을 대폭 단축한다.
- 개인정보 유출/침해사고 대응에 필요한 체크리스트를 사건 흐름 안에 통합한다.

### 3.2 성공 지표
- 사건 1건당 초동 정리 시간: 2시간 → 15분 이하
- 내부 귀속 A/B 등급 사건의 자동 매핑 성공률: 85% 이상
- 사건별 증거 패키지 생성 시간: 60초 이하
- 자동 생성 문서의 수작업 보정률: 30% 이하
- 모든 산출물에 대한 감사로그 누락률: 0%

## 4. 비목표(Non-goals)

- 외부 공인 IP의 실명 자동 식별
- 통신사/플랫폼 가입자 정보 조회 자동화
- 공격/보복/스캐닝/익스플로잇 기능
- 무인 자동 신고/무인 자동 고소
- 사람 검토 없는 법률 판단 확정
- EDR/CMDB/VPN 제품 자체를 대체하는 것

## 5. 범위

### 5.1 In Scope
- 로그 수집/정규화
- 사건 검색/상관분석
- 내부 귀속 엔진
- 외부 사건의 보수적 문서화
- 증거 고정(freeze), 해시, 감사로그
- 사건 관리 및 승인 흐름
- 사건 요약서/증거목록/고소장 초안 생성
- 법무/개인정보 대응 체크리스트

### 5.2 Out of Scope
- 네트워크 포렌식 전문도구 대체
- 메모리 포렌식/악성코드 샌드박스
- 수사기관 API 연계
- 자동 압수수색/통신자료 요청 절차 대행
- 차단/대응 SOAR 전체 기능

## 6. 사용자 및 페르소나

### 6.1 보안 분석가(SOC/IR)
- 사건을 빠르게 찾고, 관련 로그를 묶어 보고 싶다.
- IP 하나로 단말/사용자를 연결하고 싶다.
- 증거 패키지를 클릭 몇 번으로 만들고 싶다.

### 6.2 보안팀 리드/관리자
- 사건 우선순위를 보고 싶다.
- 귀속 신뢰도가 낮은 사건과 높은 사건을 구분하고 싶다.
- 대시보드와 SLA를 보고 싶다.

### 6.3 법무/대외대응 담당
- 사건 요약과 증거 목록이 정리된 문서를 받고 싶다.
- 외부 사건에서 과도한 단정 문구가 들어가지 않길 원한다.
- 검토/승인 흔적이 남길 원한다.

### 6.4 개인정보보호 담당자
- 개인정보 유출 여부 판단을 위한 사실관계를 빠르게 정리하고 싶다.
- 신고/통지 여부를 판단할 수 있는 체크리스트가 필요하다.
- 접속기록 보관, 열람통제, 증거보관 체계가 필요하다.

### 6.5 IT 자산/인프라 관리자
- 호스트명, 자산번호, 사용자 배정 정보가 정확히 연결되길 원한다.
- DHCP/NAT/VPN/EDR/CMDB 연동이 안정적으로 동작하길 원한다.

## 7. 핵심 사용자 시나리오

### 시나리오 A: 내부 자산에서 이상 접근 발생
1. 조사자가 IP 또는 호스트명 검색
2. 시스템이 DHCP/EDR/AD/VPN/CMDB를 교차 조회
3. 해당 단말, 자산번호, 로그인 계정, 주사용자 후보를 제시
4. 관련 웹/WAF/DB 로그를 타임라인으로 묶음
5. 사건 생성 및 증거 freeze
6. 사건 요약서와 고소장 초안 생성
7. 법무/보안 승인 후 외부 제출 여부 결정

### 시나리오 B: 외부 공인 IP에서 반복적인 이상 요청
1. 조사자가 IP 검색
2. 시스템이 ASN/클라우드/VPN/Tor 여부와 요청 패턴을 표시
3. 동일 IP/UA/쿠키/세션/경로 패턴과 연계
4. 사건 생성
5. 피행위자를 `성명불상`으로 표기
6. 수사기관 제출용 패키지와 고소장 초안을 생성하되, 추가 조회 필요 상태를 표시

### 시나리오 C: 개인정보 유출 가능 사건
1. 데이터 조회/다운로드 로그와 권한변경 로그를 상관분석
2. 유출 가능 범위와 대상 시스템을 사건 화면에 표시
3. 신고/통지 여부 판단용 체크리스트를 제공
4. 사건 산출물에 필요한 근거를 포함

## 8. 제품 원칙

1. **귀속은 증거 기반으로만 수행한다.**
2. **외부 실명 특정은 하지 않는다.**
3. **자동 문서는 초안일 뿐이며 반드시 승인 단계를 거친다.**
4. **원본 증거와 해시 없는 산출물은 신뢰하지 않는다.**
5. **모든 조회/수정/내보내기는 감사로그에 남는다.**
6. **민감정보는 최소한으로 표시하고 기본 마스킹한다.**

## 10. 기능 요구사항

### 10.1 수집/정규화

**FR-001** 시스템은 다중 로그 소스를 커넥터 방식으로 수집해야 한다.  
**FR-002** 시스템은 원본 로그(raw artifact)를 변경 불가능한 형태로 저장해야 한다.  
**FR-003** 시스템은 각 raw artifact에 SHA-256 해시를 계산해 저장해야 한다.  
**FR-004** 시스템은 모든 로그를 Canonical Event 스키마로 정규화해야 한다.  
**FR-005** 시스템은 이벤트 수집 시 소스, 수집시각, 파서 버전을 기록해야 한다.

### 10.2 검색/상관분석

**FR-006** 사용자는 IP, 계정, 이메일, PC명, 자산번호, 세션ID, 도메인, URL 경로로 검색할 수 있어야 한다.  
**FR-007** 검색 결과는 시간순 타임라인, 관련 엔티티, 위험도, 귀속 신뢰도를 함께 제공해야 한다.  
**FR-008** 시스템은 동일 IP, 동일 계정, 동일 세션, 동일 단말, 동일 UA 패턴을 기준으로 사건 후보를 묶어야 한다.  
**FR-009** 시스템은 다중 소스 이벤트를 1개 사건 화면에서 통합 조회할 수 있어야 한다.

### 10.3 귀속 엔진

**FR-010** 시스템은 내부 IP에 대해 DHCP lease, NAT, VPN, EDR, AD, CMDB를 상관분석해야 한다.  
**FR-011** 시스템은 내부 사건에 대해 `관측 IP -> 단말 -> 로그인 계정 -> 사용자` 순으로 연결 후보를 산출해야 한다.  
**FR-012** 시스템은 귀속 신뢰도를 A/B/C/D 등급으로 계산해야 한다.  
**FR-013** 실명 자동 기입은 A/B 등급에 한해 허용해야 한다.  
**FR-014** 공용PC, 점프서버, NAT 뒤 다중 사용자 환경에서는 다중 후보와 낮은 신뢰도를 표시해야 한다.  
**FR-015** 외부 공인 IP 사건은 기본적으로 `성명불상` 상태로 처리해야 한다.

### 10.4 탐지/알림

**FR-016** 시스템은 규칙 기반 이상행위 탐지를 지원해야 한다.  
**FR-017** 고위험 규칙 일치 시 알림 또는 사건 후보를 생성해야 한다.  
**FR-018** 탐지 결과에는 규칙명, 점수, 근거 이벤트를 포함해야 한다.

### 10.5 사건관리

**FR-019** 사용자는 알림을 사건으로 승격(promote)할 수 있어야 한다.  
**FR-020** 사건은 상태(New, Triaged, Investigating, Review, Ready for Export, Closed)를 가져야 한다.  
**FR-021** 사건에는 담당자, 우선순위, 태그, 메모, 관련 엔티티를 저장할 수 있어야 한다.  
**FR-022** 사건 화면에서 핵심 요약, 타임라인, 귀속 근거, 증거 목록을 확인할 수 있어야 한다.

### 10.6 증거관리

**FR-023** 사용자는 사건별로 증거 freeze를 실행할 수 있어야 한다.  
**FR-024** freeze 시 시스템은 관련 raw artifact, 정규화 이벤트 스냅샷, 해시 목록을 고정해야 한다.  
**FR-025** 증거 내보내기(export)는 별도 기록과 해시를 남겨야 한다.  
**FR-026** 증거 열람/다운로드 역시 감사로그에 남겨야 한다.

### 10.7 문서 생성

**FR-027** 시스템은 사건 요약서를 생성해야 한다.  
**FR-028** 시스템은 증거목록/해시 목록 문서를 생성해야 한다.  
**FR-029** 시스템은 고소장 초안을 생성해야 한다.  
**FR-030** 문서 생성은 구조화된 사건 snapshot을 기반으로 템플릿 렌더링 방식으로 수행해야 한다.  
**FR-031** 문서에는 생성시각, 사건번호, 귀속 신뢰도, 검토 상태를 표시해야 한다.  
**FR-032** 외부 사건의 고소장 초안은 `성명불상` 문구를 기본값으로 사용해야 한다.

### 10.8 권한/감사

**FR-033** 시스템은 역할 기반 접근통제(RBAC)를 제공해야 한다.  
**FR-034** 문서 승인 권한은 일반 조사자와 분리되어야 한다.  
**FR-035** 모든 생성/수정/승인/내보내기 행위는 감사로그를 남겨야 한다.  
**FR-036** 민감정보는 기본적으로 마스킹 표시해야 한다.

### 10.9 운영/관리

**FR-037** 관리자는 로그 소스별 수집상태를 확인할 수 있어야 한다.  
**FR-038** 관리자는 규칙을 버전 관리할 수 있어야 한다.  
**FR-039** 관리자는 보관기간(retention)과 보존정책을 관리할 수 있어야 한다.  
**FR-040** 시스템은 접속기록 및 내부 조사 행위에 대한 자체 감사로그를 보관해야 한다.

## 11. 비기능 요구사항

### 11.1 성능
- 단일 IP 검색 첫 응답 3초 이내(최근 7일 범위)
- 사건 상세 로딩 5초 이내
- 증거 freeze 60초 이내(평균 사건)
- 문서 생성 60초 이내

### 11.2 가용성

- 역할별 최소권한
- 비밀키/연결정보는 시크릿 관리
- 문서/증거 무결성 보장

### 11.4 감사 가능성
- 사건·증거·문서·승인·내보내기 전 단계 감사로그
- 이벤트 정규화 파서 버전 추적
- 문서 생성 입력 스냅샷 보관

## 12. 귀속 모델

### 12.1 내부 귀속
내부 귀속은 아래 경로를 따른다.

`관측 IP -> DHCP/NAT -> MAC/포트 -> 단말(EDR/CMDB) -> 로그인 계정(AD/SSO/VPN) -> 사용자`

### 12.2 외부 귀속
외부 귀속은 아래 수준까지만 수행한다.

- ASN/통신사/호스팅 사업자
- 국가/지역 추정(가능한 경우)
- VPN/프록시/Tor/클라우드 대역 여부
- 동일 IP/UA/세션/요청패턴 반복성

외부 귀속은 **실명 특정이 아니라 수사용 귀속 후보 정리**가 목표다.

### 12.3 귀속 신뢰도 등급
- **A**: DHCP + EDR + AD 로그인 + 자산관리 일치
- **B**: VPN + 디바이스 식별 + 계정로그 일치
- **C**: 일부 근거만 존재하거나 공용PC/점프서버 환경
- **D**: 외부 공인 IP만 존재하거나 근거 불충분

## 13. 문서 산출물

### 13.1 사건 요약서

## 19. 수용 기준(Acceptance Criteria)

### MVP 수용 기준
- IP 검색 시 관련 사건 후보, 이벤트 타임라인, 연결된 자산/사용자 후보가 보인다.
- 사건 생성 후 증거 freeze가 가능하다.
- freeze 결과로 해시 목록이 생성된다.
- 사건 요약서가 생성된다.
- 모든 중요 행위가 감사로그에 남는다.
- 외부 사건은 실명 없이 `성명불상`으로 출력된다.

### R2 수용 기준
- 내부 사건 A/B 등급 귀속이 가능하다.
- 고소장 초안이 템플릿으로 생성된다.
- 승인 흐름이 동작한다.
- 문서별 생성 snapshot을 조회할 수 있다.

## 20. 의사결정 메모

- 외부 공인 IP의 실명 추정 기능은 제품에서 제공하지 않는다.
- 자동 제출보다 증거 정리와 초안 품질을 우선한다.
- 문서 생성은 구조화 데이터 + 템플릿 엔진 중심으로 설계한다.
- OpenSearch는 검색/상관분석, PostgreSQL은 사건/증거/문서/감사로그의 정본(Source of Truth)으로 사용한다.
